## Providing Feedback

### General Feedback

We welcome all forms of feedback, including:

- Suggestions for improvements
- Comments on usability
- Ideas for new features
- Thoughts on documentation quality
- Performance observations

### How to Provide Feedback

1. **GitHub Discussions**: For general feedback and community discussions
   - Visit our [Discussions page](https://github.com/your-org/WDAC-Enterprise-Security/discussions)
   - Create a new discussion or join existing ones
   - Tag discussions appropriately (idea, question, show and tell, etc.)

2. **Email**: For private feedback
   - Send to: feedback@wdac-toolkit.org
   - Include "WDAC Toolkit Feedback" in the subject line
   - Provide detailed information about your experience

3. **Surveys**: Periodic surveys to gather structured feedback
   - Watch for announcements in release notes
   - Surveys typically take 5-10 minutes to complete
   - Results are shared with the community

## Reporting Issues

### Before Reporting

1. Check existing issues to avoid duplicates
2. Ensure you're using the latest version
3. Try to reproduce the issue consistently

### How to Report Issues

1. **GitHub Issues**: For technical issues and bugs
   - Go to [Issues page](https://github.com/your-org/WDAC-Enterprise-Security/issues)
   - Click "New Issue"
   - Select the appropriate issue template
   - Fill in all required information

2. **Issue Templates**:
   - Bug Report: For technical issues
   - Documentation Issue: For documentation problems
   - Security Vulnerability: For security concerns (see separate section)

### What to Include in Bug Reports

- Clear, descriptive title
- Steps to reproduce the issue
- Expected vs. actual behavior
- Screenshots or videos if applicable
- System information (OS version, PowerShell version, etc.)
- Error messages and logs
- Workarounds you've tried

### Security Vulnerabilities

For security-related issues:

- Email: security@wdac-toolkit.org
- Do not publicly disclose security issues
- Follow responsible disclosure practices
- See our [Security Policy](SECURITY.md) for more details

## Requesting Features

### Feature Request Process

1. **Check Existing Requests**: Search issues and discussions
2. **Create a Request**: Use the Feature Request template
3. **Provide Details**: Explain the use case and benefits
4. **Community Input**: Engage with others who might want the feature
5. **Prioritization**: Team reviews and prioritizes requests

### What Makes a Good Feature Request

- Clear problem statement
- Proposed solution or approach
- Use cases and benefits
- Potential impact on users
- Any alternatives considered
- Willingness to contribute (if applicable)

### Feature Prioritization Factors

Features are prioritized based on:

- Community demand and votes
- Alignment with project goals
- Implementation complexity
- Resource availability
- Strategic importance
